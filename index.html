CASE 01
백그라운드 동기화 — WorkManager를 선택한 이유
AlarmManager, JobScheduler, WorkManager 중 왜 WorkManager인가
문제

백그라운드에서 서버와 데이터를 주기적으로 동기화해야 했습니다. 앱이 종료되거나 기기가 재시작되어도 작업이 보장되어야 했습니다.

결정

AlarmManager는 Doze Mode(Android 6+)에서 throttle됩니다. JobScheduler는 Android 5.0 미만 미지원 + API가 복잡합니다. WorkManager는 내부적으로 상황에 맞는 구현체를 선택하며, 작업 지속성(앱 종료 후에도 재시작)을 보장합니다.

결과

배터리 최적화 모드에서도 동기화 성공률 97.3% → 99.8%로 개선. 앱 재설치 후 작업 재등록 로직 제거로 코드 50라인 감소.

동기화 성공률 +2.5% 코드 50라인 감소 Android 5.0+ 단일 구현
SyncWorker.kt
class SyncWorker(
    context: Context,
    params: WorkerParameters
) : CoroutineWorker(context, params) {

    override suspend fun doWork(): Result {
        return try {
            val newData = apiService.fetchLatest()
            database.upsert(newData)
            Result.success()
        } catch (e: Exception) {
            // 3회 재시도 후 실패 처리
            if (runAttemptCount < 3) Result.retry()
            else Result.failure()
        }
    }
}

// 등록: 15분 간격, 네트워크 연결 시에만 실행
val syncRequest = PeriodicWorkRequestBuilder<SyncWorker>(15, TimeUnit.MINUTES)
    .setConstraints(
        Constraints.Builder()
            .setRequiredNetworkType(NetworkType.CONNECTED)
            .build()
    )
    .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
    .build()

WorkManager.getInstance(context)
    .enqueueUniquePeriodicWork("sync", ExistingPeriodicWorkPolicy.KEEP, syncRequest)
CASE 02
앱 시작 시간 3.2초 → 0.8초 (75% 개선)
Cold start 성능 최적화 — 측정부터 개선까지의 과정
문제

Cold start 시간이 3.2초. 사용자 리뷰에 "느리다"는 피드백이 반복됨. Android Vitals 기준(2초) 초과로 스토어 노출에도 불이익.

결정

Perfetto로 측정 → Application.onCreate()에서 불필요한 SDK 초기화 4개 발견. App Startup 라이브러리로 지연 초기화 적용, Hilt 초기화 순서 최적화, SplashScreen API 도입으로 화면 전환 제거.

결과

Cold start 3.2초 → 0.8초. Android Vitals "좋음" 구간 진입. 1주 후 스토어 노출 지수 18% 상승.

시작 시간 75% 단축 3.2s → 0.8s 스토어 노출 +18%
AppStartupInitializer.kt
// App Startup으로 초기화 순서 제어
class AnalyticsInitializer : Initializer<Unit> {
    override fun create(context: Context) {
        // 메인스레드 블로킹 없이 백그라운드 초기화
        CoroutineScope(Dispatchers.IO).launch {
            FirebaseAnalytics.getInstance(context)
        }
    }
    // Hilt 초기화 이후 실행 보장
    override fun dependencies() = listOf(HiltInitializer::class.java)
}

// Perfetto 측정으로 발견한 병목 — 개선 전
class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()
        HeavySdkA.init(this)   // 800ms — 제거
        HeavySdkB.init(this)   // 600ms — 지연 초기화로 전환
        HeavySdkC.init(this)   // 400ms — 제거
    }
}
CASE 03
MVVM vs MVI — 아키텍처 선택의 트레이드오프
팀 규모와 앱 복잡도에 따른 아키텍처 결정 과정
문제

2인 팀에서 5인 팀으로 확장되면서 상태 관리 충돌 증가. UI 상태가 여러 곳에서 변경되어 디버깅이 어려워짐.

결정

MVI 패턴 도입. Intent → State → View 단방향 데이터 흐름으로 상태 변경 지점을 ViewModel 단일 위치로 수렴. 다만 보일러플레이트 증가는 인정하고 팀과 합의.

결과

상태 관련 버그 64% 감소. 신규 기능 개발 시 예측 가능성 향상. 온보딩 시간 2주 → 3일로 단축.
